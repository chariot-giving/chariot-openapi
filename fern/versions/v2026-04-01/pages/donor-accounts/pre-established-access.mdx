title: "DAF-Initiated Setup"
subtitle: "DAF creates an Account and issues an Authorization Token before the donor ever opens DAFpay."

---

In this flow, donors must opt in to DAFpay use via their DAF's portal **before** they can submit grants on a nonprofit's donation form. The DAF creates the Account and an Authorization Token in advance, surfaces the code to the donor, and the donor enters the code into DAFpay during profile setup. This auto-approves the Account at the moment of first use — the DAF has zero follow-up verification work.

## When to use this flow

This flow is well-suited to DAFs that:

- **Want explicit donor consent** before grants can be submitted.
- **Don't want to support "give at the point of inspiration"** — every donor must complete the DAF's opt-in flow first.
- **Already have a portal** where donors can authenticate and enable third-party services.

## Step-by-step

### 1. Add an "Enable DAFpay Access" entry point in your portal

This is the donor-facing entry point. The simplest implementation is a button on the donor's account page; clicking it kicks off your portal-side handler.

If your donors can have multiple Giving Pools, your handler may also include a UI for the donor to choose which Giving Pool to enable. The Giving Pool choice maps to the `giving_pool_id` you'll send to Chariot.

### 2. Create the Donor Account

Call [Create Donor Account](/api/donor-accounts/create) with the donor's email and (recommended) your internal `external_id`. The email **must** match the email the donor will use when they authenticate with DAFpay — DAFpay matches incoming verified emails to existing Accounts.

<CodeBlocks>
```bash cURL
curl -X POST https://api.givechariot.com/v1/donor_accounts \
  -H "Authorization: Bearer $CHARIOT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "giving_pool_id": "0bf40881-8ee2-47fb-98ca-f58c7999aa34",
    "email": "warrenBuffet@example.com",
    "external_id": "ACME-DAF-DONOR-1042",
    "first_name": "Warren",
    "last_name": "Buffet"
  }'
```
```typescript TypeScript
const account = await chariot.donorAccounts.create({
  giving_pool_id: "0bf40881-8ee2-47fb-98ca-f58c7999aa34",
  email: "warrenBuffet@example.com",
  external_id: "ACME-DAF-DONOR-1042",
  first_name: "Warren",
  last_name: "Buffet",
});
```
</CodeBlocks>

<Note title="Idempotency on `(giving_pool_id, email)`">
A given `(giving_pool_id, email)` pair can only have one Account. If the donor previously enabled access and clicks the button again, the second create call returns `409 Conflict`. Use [List Donor Accounts](/api/donor-accounts/list) with `email=...` to look up the existing account and create a new token instead.
</Note>

### 3. Create an Authorization Token

Call [Create Authorization Token](/api/authorization-tokens/create) on the new Account.

<CodeBlocks>
```bash cURL
curl -X POST https://api.givechariot.com/v1/donor_accounts/donor_account_01jpjenf5q6cawy43yxfcrxhct/authorization_tokens \
  -H "Authorization: Bearer $CHARIOT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{}'
```
```typescript TypeScript
const token = await chariot.authorizationTokens.create(
  "donor_account_01jpjenf5q6cawy43yxfcrxhct",
  {}
);
```
</CodeBlocks>

### 4. Surface the code to the donor

The simplest pattern is to display the code inline on the donor's portal page after they click "Enable DAFpay Access". Pair it with brief instructions:

> Your DAFpay code is **DAFP-7K3X-9M4Q**.
>
> Copy this code and paste it into the DAFpay form on any participating nonprofit's donation page when prompted to verify your account. The code is single-use and expires in 30 days.

If the donor loses the code before using it, give them a "Generate a new code" action that revokes the existing token and creates a new one.

### 5. Donor enters the code in DAFpay

When the donor opens DAFpay on a nonprofit's donation form, authenticates with their email, and reaches profile setup, they will be prompted to enter the code (DAFpay detects that the email matches an Account in `pending` status with at least one valid Authorization Token).

When they enter the code, DAFpay verifies it client-side, the Account transitions to `approved`, and the donor proceeds directly to submitting a Grant Request.

### 6. You receive a webhook

You'll receive a `donor_account.updated` event — fetch the Donor Account to confirm its current status. Subsequent `grant.created` events for this donor flow through normally — no further verification work required.

## Token lifecycle considerations

- **Default expiration**: 30 days. Configurable via `expires_in` on token creation (60s — 90 days).
- **Single-use**: Once consumed, a token is `consumed` and cannot be re-verified.
- **Revocation**: If a donor loses their code, [revoke](/api/authorization-tokens/revoke) the existing token and create a new one. Don't have two valid tokens active at once.

## Comparing to Donor-Initiated Verification

| | Donor-Initiated Verification | DAF-Initiated Setup |
|---|---|---|
| Who initiates? | Donor (submits grant first) | DAF (creates Account in portal first) |
| Give at point of inspiration? | Yes | No — donor must pre-register |
| Donor opt-in required up front? | No | Yes |
| DAF work per new donor | Single API call after code is received | Create Account + token in advance |
| DAF work per Grant Request | Approve grant request | Approve grant request |
| Best for | Light-touch setup, give-first flows | Portal-gated access, explicit consent flows |
